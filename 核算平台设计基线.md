# 核算平台设计基线

## 目标

本基线用于统一 `cost_platform` 的产品设计、数据设计和代码实现标准。

它服务的不是“临时可用的后台页面”，而是一套可长期演进、可审计、可追溯、可多场景复用的企业级成本核算平台。

## 一、产品定位

### 1. 平台定位

核算平台不是单一费用工具，而是面向企业级业务配置和运行治理的平台型产品。

平台至少要支持：

- 多业务域
- 多场景
- 多费用
- 多规则
- 多版本
- 可追溯运行

### 2. 用户定位

主要面向三类用户：

- 业务配置人员：维护场景、费用、规则和阶梯
- 管理与审核人员：关注发布、变更、审计和差异
- 核算执行人员：关注试算、正式核算、批量任务和结果追溯

### 3. 建设原则

- 先把模型做清楚，再叠加复杂功能
- 先把维护路径收顺，再扩展治理能力
- 先统一共性能力，再做页面个性化表达

## 二、核心对象

### 1. 业务域

业务域用于表达一个场景属于哪类业务，例如：

- 薪资核算
- 劳务结算
- 港口作业
- 仓储结算
- 运输计费

业务域是分类标签，不直接参与金额计算。

### 2. 场景

场景是核算平台的第一层业务组织边界，可理解为：

- 合同
- 核算主题
- 业务方案
- 公司级核算域

场景负责承载后续所有费用、变量、规则和发布版本。

### 3. 费用

费用是场景下需要核算的具体费用项，例如：

- 底薪
- 夜班补贴
- 装卸作业费
- 管理费

费用是规则维护的直接承接层。

### 4. 变量

变量是费用判断和计算依赖的影响因素，可来源于：

- 字典下拉
- 接口下拉
- 数值输入
- 文本输入
- 公式派生

变量不是某一条规则的私有字段，而是平台的公共计算要素。

### 5. 规则

规则是某个费用在不同条件下的取价方式。

规则至少要回答三个问题：

- 什么条件下命中
- 命中后取什么费率或公式
- 是否存在优先级或互斥关系

### 6. 阶梯

阶梯是规则下更细一层的费率明细。

只有当规则计价方式为阶梯时，才需要维护阶梯明细。

### 7. 发布版本

发布版本是场景级的可运行快照，不是单个页面状态。

它的作用是：

- 固化运行时依赖的整套配置
- 保证结果可复现
- 支持追溯、审计和回滚

## 三、产品主线

成熟核算平台的第一条主线应固定为：

`业务域 -> 场景 -> 费用 -> 规则 -> 阶梯 -> 发布`

对应到页面维护路径，应尽量让业务人员按这个顺序理解和操作。

### 当前重构顺序

第一阶段：

- 系统字典
- 场景中心
- 费用中心
- 规则中心
- 阶梯规则

第二阶段：

- 发布中心
- 版本差异
- 引用校验
- 删除保护

第三阶段：

- 试算
- 正式核算
- 批量任务
- 结果追溯

## 四、发布边界

### 1. 为什么按场景发布

发布粒度按场景，而不是按单个费用。

原因是运行时真正依赖的是一整套配置快照，包括：

- 场景本身
- 费用
- 变量
- 规则
- 阶梯
- 字典和配置口径

如果只追费用本身，无法保证结果完全复现。

### 2. 为什么又要支持费用级追溯

业务人员理解结果时，通常不是按场景整体理解，而是按某个费用理解。

所以平台必须同时支持：

- 场景级发布
- 费用级查看和解释

一句话：

场景级发布是为了运行稳定，费用级追溯是为了业务可理解。

### 3. 删除与停用控制

判断对象是否可删，依据不是“是否发布过”，而是：

- 是否仍被规则引用
- 是否仍被有效版本引用

正确规则应是：

- 未被引用：可删
- 被草稿引用：提示后处理
- 被已发布版本引用：阻止直接删除，并提示先解除引用再发布新版本

## 五、页面设计基线

### 1. 页面必须贴业务维护路径

页面设计不能只站在开发视角，而要贴合业务人员心智。

优先采用：

- 先选场景
- 再看费用
- 再维护规则
- 最后维护阶梯

### 2. 通用能力优先抽象

以下能力默认要抽公共层，不应每页重复写一套：

- 页面导语与引导卡片
- 场景远程下拉
- 业务域字典下拉
- 状态字典下拉
- 查询栏基础布局
- 标准 CRUD 弹窗模式

### 3. 录入方式要适合业务

如果某类配置天然是批量组合录入，后续应优先考虑：

- 可编辑表格
- 复制并改值
- 下拉自动带出
- 条件值批量改写

不应让业务人员长期依赖大段长表单重复录入。

## 六、数据设计基线

### 1. 表设计

- 核算业务表统一使用 `cost_` 前缀
- 所有表必须有表注释
- 所有字段必须有字段注释
- 编码字段必须明确唯一性边界

### 2. 基础字段建议

常见业务表建议统一考虑：

- 主键
- 所属场景主键 / 编码
- 业务编码
- 状态
- 备注
- 创建人
- 创建时间
- 更新人
- 更新时间

### 3. 约束设计

必须提前考虑：

- 唯一约束
- 引用关系
- 状态语义
- 删除边界
- 版本快照承接方式

## 七、代码实现基线

### 1. 后端

后端遵循若依标准分层：

- controller
- service
- mapper
- domain

但不能只停留在生成代码水平。

所有关键链路要补充中文注释，至少写清：

- 这段逻辑的业务目的
- 它和上游/下游的关系
- 后续预留的扩展点

### 2. 前端

前端应遵循：

- 通用组件优先
- 通用 API 优先
- 查询模型尽量统一
- 页面表达业务化

禁止为了赶进度复制整页再改字面量。

### 3. 文案

文案要以业务易懂为优先。

例如：

- “价格策略”可逐步收敛为“费率规则”
- “场景”要配说明文案帮助理解为合同/主题/方案
- “发布”要明确解释为运行版本快照

## 八、当前执行要求

从现在开始，后续功能改造要同时满足：

- 符合若依项目开发规范
- 符合 `docs/核算模块开发规范.md`
- 符合本设计基线

当三者冲突时，优先保证：

1. 企业级核算产品建模正确
2. 业务人员可理解、可维护
3. 若依规范化实现
