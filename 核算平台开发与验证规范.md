# 核算平台开发与验证规范

## 1. 文档目标

本规范用于约束 `cost_platform` 的后续开发过程，确保：

- 开发按统一设计执行
- 代码风格一致
- 数据设计可落地
- 验证链路完整
- 每次交付都可回归、可审计、可追溯

## 2. 开发前必做事项

每次新增模块或重大改造前，必须先完成以下检查：

### 2.1 设计对齐

必须先对照以下文档：

- `docs/核算平台设计基线.md`
- `docs/企业级核算平台详细设计.md`
- `docs/核算模块开发规范.md`
- `docs/核算平台缓存设计与Redis规范.md`
- `D:/Desktop/charge/docs/platform/产品计划清单.csv`

如果当前需求与文档冲突，先更新文档，再开发。

此外，每轮重大开发前需同步确认：

- `charge` 最新成熟版本中已经落下来的功能是否需要吸收
- `charge` 最新页面设计思想是否已纳入当前实现
- 清单中的高优先级治理项是否需要前移

### 2.2 业务边界确认

开发前必须明确：

- 该功能位于哪一层：场景、费用、变量、规则、阶梯、发布、执行、追溯
- 它的上游对象是谁
- 它的下游对象是谁
- 是否会影响发布链路
- 是否需要引用校验
- 是否需要缓存或 Redis 锁

## 3. 前端开发规范

### 3.1 页面结构规范

核算模块页面优先采用统一工作台结构：

1. 导语卡
2. 顶部指标卡片
3. 检索面板
4. 操作栏
5. 主表格/工作区
6. 弹窗或分栏明细

### 3.2 组件复用规范

以下能力优先提取共享组件，不允许整页复制：

- 顶部导语卡
- 指标卡片条
- 检索面板
- 操作栏
- 场景远程下拉
- 业务域下拉
- 状态下拉
- 通用表格操作列
- 阶梯编辑器
- 条件值编辑器

### 3.3 样式规范

- 优先复用若依和核算模块共享样式
- 避免行内样式
- 避免每个页面单独写一套相同样式
- 页面特殊样式必须说明原因
- 组件粒度尽量和业务对象对齐

### 3.4 文案规范

- 优先使用业务人员易懂表达
- 避免偏技术化术语直接暴露
- 复杂概念必须配说明文案

### 3.5 页面数据源规范

- 页面列表、详情、下拉和统计卡片默认直接查询后台接口
- 需要缓存时必须通过后端统一接入 Redis 或共享缓存，前端只消费接口结果
- 禁止新增前端本地 store、localStorage、sessionStorage 作为字典、主数据和治理状态的事实来源
- 前端内存仅用于当前页面渲染态，不作为多人协作场景下的一致性依据

## 4. 后端开发规范

### 4.1 分层规范

后端严格遵循若依项目分层：

- Controller
- Service
- Mapper
- Domain

### 4.2 链路注释规范

以下代码必须写中文链路注释：

- 发布
- 生效切换
- 版本快照
- 删除保护
- 引用校验
- 试算
- 正式核算
- Redis 锁与幂等

### 4.3 校验规范

必须显式实现：

- 基础字段校验
- 唯一性校验
- 引用关系校验
- 状态合法性校验
- 发布前校验

### 4.4 查询规范

- 列表接口以分页为主
- 下拉远程搜索接口返回轻量摘要
- 不能把完整业务对象用于轻量场景选择

### 4.5 MyBatis-Plus 使用规范

- 后端持久层采用 MyBatis-Plus 渐进集成策略，不做一次性全量推翻
- 新增或重构的核算模块，单表 CRUD 和条件构造优先使用 `BaseMapper` 与 Lambda Wrapper
- 复杂统计、治理校验、跨表分析 SQL 继续保留 XML Mapper
- 使用 `PageHelper` 的旧分页链路保持不动；新增 MP 分页时不要与 `PageHelper` 同时作用在同一查询上
- 基础审计字段沿用若依原生 `BaseEntity`，新 MP 模块通过自动填充统一写入创建人、修改人、创建时间、修改时间

## 5. 数据库开发规范

### 5.1 建表前

建表前必须明确：

- 表属于哪条业务主线
- 与哪些表有关联
- 哪些字段需要唯一索引
- 哪些字段需要查询索引
- 是否需要逻辑删除
- 是否进入发布快照

### 5.2 建表要求

- 统一使用 `cost_` 前缀
- 所有表必须有表注释
- 所有字段必须有字段注释
- 字段命名要有业务语义
- 索引要能解释用途

### 5.3 变更脚本要求

- 初始化脚本和增量脚本分离
- 每次变更要写清楚适用版本和变更目的
- 脚本必须支持顺序执行

## 6. Redis 与缓存开发规范

- 严格遵循 `docs/核算平台缓存设计与Redis规范.md`
- 先定义缓存 Key 再写代码
- 发布、生效、正式核算必须明确锁和幂等
- 不允许把 Redis 当最终数据源

## 7. 测试与验证规范

### 7.1 前端验证

每次前端改动至少验证：

- 页面是否正常打开
- 查询、重置、分页是否正常
- 新增、修改、删除、导出是否正常
- 共享组件是否影响其他页面
- 深色主题和浅色主题是否可用
- 不同分辨率是否正常
- 页面刷新后是否仍以后台或 Redis 共享缓存为准，不依赖本地缓存清理

### 7.2 后端验证

每次后端改动至少验证：

- 项目是否能正常编译
- 关键接口是否能正常调用
- 分页、查询、保存、删除是否符合预期
- 校验逻辑是否生效
- 错误提示是否可理解
- 若本次引入 MyBatis-Plus，需验证旧 XML Mapper 与新 MP Mapper 能共存运行

### 7.3 数据验证

每次涉及数据库改动至少验证：

- 建表脚本可执行
- 注释完整
- 索引正确
- 初始化数据可重复执行或有清晰执行顺序

### 7.4 缓存验证

每次涉及 Redis 改动至少验证：

- Key 命名是否符合规范
- 命中与回源逻辑是否正确
- 失效是否及时
- 锁是否可释放
- 幂等是否生效

### 7.5 性能验证

涉及正式核算、批量任务、结果台账时，至少验证：

- 批量执行是否采用异步任务模式
- 结果写入是否支持批量提交
- 列表是否分页查询
- 查询是否命中预期索引
- 运行时是否优先复用发布快照装配结果
- 月近 100 万条规模下是否仍可支撑正常计费和结果查询

## 8. 每次交付的最小验收清单

每个可交付功能都必须至少满足：

- 设计文档已覆盖
- 前端页面可用
- 后端接口可用
- SQL 脚本可执行
- 关键链路有中文注释
- 构建通过
- 关键交互已手动验证

## 9. 推荐验证命令

前端：

- `pnpm install`
- `npm run build:prod`

后端：

- `mvn -q -DskipTests package`

数据库：

- 执行初始化脚本
- 执行增量脚本

## 10. 缺陷处理规范

出现问题时优先级如下：

1. 先判断是模型问题还是实现问题
2. 若是模型问题，先修文档
3. 若是实现问题，先补共享层，再修页面
4. 修复后必须重新验证构建和关键链路

## 11. 执行约定

从现在开始，后续开发默认遵循：

- 先设计，后开发
- 先共享，后个性
- 先验证，后提交
- 先说明边界，后扩功能
