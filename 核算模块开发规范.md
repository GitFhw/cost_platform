# 核算模块开发规范

## 适用范围

本规范适用于 `cost_platform` 项目中的核算业务模块，包括但不限于：

- 配置中心
- 系统字典扩展
- 变量中心
- 费用主数据
- 费率规则
- 阶梯规则
- 发布与追溯

## 数据库命名规范

### 表命名

- 核算模块所有业务表必须统一使用 `cost_` 前缀
- 表名应体现业务语义，避免无意义缩写

示例：

- `cost_scene`
- `cost_fee_item`
- `cost_rule`
- `cost_rule_condition`
- `cost_rule_tier`
- `cost_publish_version`

### 字段命名

- 统一使用小写字母加下划线
- 字段名优先表达业务含义
- 主键、编码、状态、时间字段命名保持一致性

示例：

- `id`
- `scene_code`
- `fee_code`
- `rule_code`
- `status`
- `created_at`
- `updated_at`

## 数据库注释规范

### 表注释

- 所有核算模块表都必须编写表注释
- 表注释要能让业务和开发一眼看出该表用途

示例：

```sql
create table cost_fee_item (
  id bigint primary key comment '主键',
  fee_code varchar(64) not null comment '费用编码'
) comment = '核算平台-费用主数据表';
```

### 字段注释

- 所有字段都必须有字段注释
- 字段注释不能只写“名称”“备注”这类低信息量描述
- 注释应说明业务含义、取值口径或用途

示例：

- `scene_code`：场景编码，用于唯一标识一个核算主题或合同场景
- `fee_code`：费用编码，用于唯一标识某个费用项
- `basis_variable_code`：阶梯依据变量编码，表示命中阶梯时使用的比较变量

## 后端代码规范

### 注释要求

- 生成或编写的后端代码必须包含清晰的中文注释
- 重点链路必须有“链路注释”，说明上下游调用关系和业务目的
- 不允许只保留工具生成的空泛注释

重点要求写注释的位置：

- Controller 接口用途与入参说明
- Service 核心流程说明
- 复杂校验逻辑
- 发布、引用校验、追溯等关键链路
- 数据转换与持久化过程

### 链路注释示例

```java
// 删除变量前先做引用校验：
// 1. 查询变量是否被费用规则引用
// 2. 查询变量是否进入已发布版本
// 3. 若仍被有效引用，则阻断删除并返回引用详情
public void removeVariable(Long variableId) {
    // ...
}
```

## 建模原则

- 先建业务主线，再补技术细节
- 优先保证业务人员易懂、易维护
- 基础字典与业务主数据分离
- 发布面向场景版本，追溯支持费用级查看

## 当前统一约定

- 前端目录：`cost_platform_front`
- 后端目录：`cost_platform_back`
- 核算业务表统一使用 `cost_` 前缀
- 所有表必须有表注释
- 所有字段必须有字段注释
- 后端关键链路必须补充中文详细注释
