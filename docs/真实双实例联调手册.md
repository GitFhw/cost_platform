# 真实双实例联调手册

## 目标

验证企业级核算平台在真实双实例部署下的基础运行能力，聚焦当前阶段已经实现的能力边界：

- 两个 `cost_admin` 实例可同时启动并连接同一套 MySQL、Redis
- 两个实例可通过 `cost.dispatch.node-id` 拥有明确、稳定的执行节点标识
- 两个实例都可以独立登录、提交正式核算任务、查询同一套任务台账
- 正式核算任务与分片会把 `execute_node` 正确落库，并在对端实例查询时保持一致
- 同一任务在 owner 节点故障后，可由另一节点完成接管并最终收尾
- 同一任务多个分片可由两个实例实时并行分担

## 前置条件

- 本地或测试环境可同时启动两个 `cost_admin` 实例
- 两个实例连接同一套 MySQL 与 Redis
- 数据库迁移已执行到 `20260409.028`
- 已存在至少一个可用场景和已发布版本
- 当前项目 jar 可在 Windows 本地正常启动

## 当前阶段标准联调步骤

1. 分别启动节点 A、节点 B。
2. 确认 `18080`、`18081` 两个端口均已监听。
3. 通过两个实例分别登录系统。
4. 使用同一场景、同一发布版本，在两个实例上分别提交一笔多分片正式核算任务。
5. 在节点 A 查询节点 B 提交的任务，在节点 B 查询节点 A 提交的任务，确认共享台账可见。
6. 查询 `cost_calc_task` 与 `cost_calc_task_partition`，确认 `execute_node` 与实例一致。
7. 再提交一笔多分片任务到节点 A，明确看到 `COST-NODE-A` 已认领并进入运行中后，强制停止节点 A。
8. 在节点 B 持续轮询任务详情，并结合数据库核查接管结果。

### 3. 同一任务跨节点实时分片协同

本次继续完成了一轮真实双实例协同验证，验证口径是：

- 任务先由节点 A 提交并成为任务 owner
- 节点 A 先认领并执行前 8 个分片
- 节点 B 在 owner 节点未宕机的情况下，实时认领剩余 `INIT` 分片
- 最终同一任务的不同分片分别由 `COST-NODE-A` 与 `COST-NODE-B` 完成

实际任务：

- `task_id = 323`
- `request_no = PERF009-REAL-CLEAN-232820`
- `bill_month = 2027-04`

- `partition 1 ~ 8 = RUNNING / COST-NODE-A`
- `partition 9 = RUNNING / COST-NODE-B`
- `partition 10 ~ 11 = INIT / 未认领`

## 当前阶段验收口径

满足以下条件即可判定当前阶段真实双实例联调通过：

- 两个实例可在同一套 MySQL、Redis 上稳定启动
- 两个实例的节点标识可区分，且任务落库 `execute_node` 与实例一致
- 两个实例都能成功提交正式核算任务
- 对端实例可查询到同一数据库中的任务与分片结果
- 任务与分片状态最终成功收尾，没有重复提交、重复落账
- 已验证“节点 A 已认领并开始执行的同一任务”，在节点 A 异常退出后可由节点 B 接管并收尾
- 已验证“节点 A 仍在线运行时，同一任务剩余分片可由节点 B 实时协同执行”
