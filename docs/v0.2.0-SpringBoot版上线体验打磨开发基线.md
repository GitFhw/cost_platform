# v0.2.0 Spring Boot 版上线体验打磨开发基线

更新时间：2026-04-15

## 1. 文档定位

本文件作为 `v0.2.0` Spring Boot 版上线前的体验打磨与开发收口基线。

它不替代以下文档：

- `核算平台设计基线.md`：回答平台长期设计原则。
- `docs/v0.2.0开发执行方案.md`：回答 0.2.0 的技术主线和执行顺序。
- `docs/统一缺陷与开发清单.md`：记录跨阶段缺陷和任务台账。
- `docs/正式核算上线最小放行条件.md`：记录正式核算链路的最小放行条件。

本文件重点回答一个更落地的问题：

当前系统已经能跑，但还不够顺、不够稳、不够像企业级产品。后续开发必须围绕哪些细节继续打磨，才能把 Spring Boot 版从“可运行”推进到“可上线、可维护、业务人员用起来得进儿”。

## 2. 目标状态

### 2.1 版本目标

`v0.2.0` Spring Boot 版上线收口目标：

- 后端可稳定启动。
- 数据库迁移可控、可追踪、可重复验证。
- 核算建模主链路顺畅：场景 -> 费用 -> 变量 -> 规则 -> 发布。
- 核算运行主链路可用：数据准备 -> 试算 -> 正式核算 -> 结果台账 -> 追溯。
- 删除、停用、发布、核算等高风险动作都有治理说明。
- 页面不只“能操作”，还要让业务人员知道当前在哪、下一步该做什么、为什么不能做。
- 错误提示不能直接暴露 SQL 堆栈，必须转成业务可理解文案。
- 上线前有清晰回归证据和数据库备份/回滚方案。

### 2.2 “得进儿”的判定标准

以下体验才算“得进儿”：

- 用户打开页面后，能看懂当前模块的用途。
- 用户完成一个动作后，系统能告诉他下一步建议。
- 用户不能操作时，系统能说明原因和解除路径。
- 用户删除、停用、发布前，能预见影响范围。
- 用户不需要理解数据库表和外键，也能判断为什么被拦住。
- 研发新增能力时，不会继续制造隐藏关系、脏数据和不可解释状态。

## 3. 优先级和状态口径

### 3.1 优先级

| 优先级 | 含义 |
| --- | --- |
| `P0` | 阻塞 Spring Boot 版上线或会造成数据错误、启动失败、核心链路不可用 |
| `P1` | 不一定阻塞启动，但明显影响业务人员使用、治理可信度或上线验收 |
| `P2` | 体验增强、效率提升、长期平台化能力 |

### 3.2 状态

| 状态 | 含义 |
| --- | --- |
| `已完成` | 已编码并通过基础验证 |
| `待验证` | 代码或方案已具备，但缺真实环境/人工回归证据 |
| `待开发` | 需要进入后续开发 |
| `持续治理` | 每轮开发都要遵守，不以一次任务关闭 |

## 9. 开发硬约束

后续任何开发必须遵守以下约束：

1. 不再只修前端报错，必须同时确认后端数据模型和治理口径。
2. 不再新增没有来源说明的关系表记录。
3. 涉及删除、停用、发布、重算的功能必须先有影响分析。
4. 涉及数据库结构变更必须同步维护上线初始化 SQL，并补充目标环境补库说明。
5. 已有库只跑增量 migration，不得重跑初始化 SQL。
6. MyBatis XML 不使用 `parameterType="java.lang.Long[]"` 这类启动期不可解析写法。
7. 后端异常不能直接把 SQL 堆栈展示给前端用户。
8. 页面新增按钮必须回答：谁用、何时用、失败后怎么办。
9. 每个核心页面必须有详情/治理入口，不能只有列表和编辑。
10. 每次完成一个能力块，必须同步更新本文件或 `docs/统一缺陷与开发清单.md`。

## 10. 建议推进顺序

### 第一轮：上线稳定性收口

目标：保证系统稳定启动、迁移正确、核心链路可跑。

优先处理：

- `P0-BOOT-*`
- `P0-DB-*`
- `P0-FLOW-*`
- `P0-DEL-*`

### 第二轮：让配置人员用得顺

目标：把场景、费用、变量、规则这条维护链路收顺。

优先处理：

- `UX-CTX-*`
- `UX-PAGE-*`
- `UX-FORM-*`
- `FEE-P1-*`
- `VAR-P1-*`
- `RULE-P1-*`

### 第三轮：让上线和运行有治理

目标：发布、试算、正式核算、结果追溯形成闭环。

优先处理：

- `UX-GOV-*`
- `PUB-P1-*`
- `RUN-P1-*`
- `RES-P1-*`

### 第四轮：产品化增强

目标：从项目可用走向平台好用。

优先处理：

- `P2-*`

## 11. 每轮开发完成定义

每轮开发完成后，必须满足：

1. 对应清单项状态已更新。
2. 后端相关模块打包通过。
3. 前端生产构建通过，若涉及前端。
4. 数据库 migration 已验证。
5. 核心页面人工操作通过。
6. 删除、停用、发布等高风险动作有治理提示。
7. 失败提示为业务文案，不暴露 SQL 堆栈。
8. 若改变业务口径，必须同步设计文档或本基线。

## 12. 当前阶段结论

当前系统已经从“能启动、能操作”进入“上线前打磨”阶段。

下一步不建议盲目新增大功能，而应优先把以下三类问题磨平：

1. 操作链路不顺：用户不知道当前在哪、下一步去哪。
2. 治理解释不足：用户知道被拦住，但不知道怎么解除。
3. 错误提示技术化：系统知道错在哪，业务人员看不懂。

只有这三类问题逐步收口，Spring Boot 版才算真正具备企业级上线基础。
