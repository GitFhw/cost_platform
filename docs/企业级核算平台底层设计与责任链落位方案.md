## 1. 文档目的

本文档用于沉淀当前企业级核算平台的底层设计方向，明确以下问题：

- 从 [formula](/D:/Desktop/formula/formula) 项目中，哪些底层思想值得借鉴。
- 责任链模式在企业级核算平台中应该放在哪些环节使用。
- 公式执行、变量解析、第三方接入、费用计算、结果落账之间如何分层。
- 当前 `cost/dev` 阶段代码在平台中的定位，以及后续演进方向。

本文档只讨论平台底层设计，不展开具体业务计费规则。

## 2. 参考项目的可借鉴点

### 2.1 表达式执行引擎

参考位置：

- [ExpressionEngine.java](/D:/Desktop/formula/formula/config/ExpressionEngine.java)
- [Formula.java](/D:/Desktop/formula/formula/calculation/Formula.java)

可借鉴点：

- 统一封装表达式引擎，而不是在业务代码里直接散落执行公式。
- 将表达式编译与表达式执行分开，避免每次运行重复解析。
- 保留“命名空间 + 自定义函数”的机制，让规则和公式在平台上具备稳定扩展能力。

不建议照搬点：

- 公式仍写在 Java 组件内部，不适合企业级可配置平台。

### 2.2 依赖递归求值与环检测

参考位置：

- [ExpressionCalculator.java](/D:/Desktop/formula/formula/calculation/ExpressionCalculator.java)

可借鉴点：

- 自动识别公式依赖项。
- 递归求值，先计算依赖项，再计算当前项。
- 对循环依赖进行显式检测。

这部分是参考项目最值得借鉴的核心能力。  
对企业级核算平台而言，这比“写死执行顺序”更稳，因为平台后续一定会出现：

- 派生变量依赖派生变量
- 费用项依赖其他费用项
- 规则条件依赖多个中间变量

### 2.3 自定义函数注册中心

参考位置：

- [ICustomFunction.java](/D:/Desktop/formula/formula/service/functions/ICustomFunction.java)
- [ExpressionEngine.java](/D:/Desktop/formula/formula/config/ExpressionEngine.java)

可借鉴点：

- 函数不是散落的工具方法，而是统一注册到公式引擎。
- 可扩展函数命名空间，例如 `fn_if`、`fn_max`、`fn_min`。

这套机制非常适合平台化，因为企业级核算中会稳定存在如下函数类型：

- 条件函数
- 空值函数
- 区间函数
- 日期函数
- 精度处理函数
- 字典映射函数
- 文本拼接函数

### 2.4 组件化计算单元

参考位置：

- [SalaryComponent.java](/D:/Desktop/formula/formula/component/SalaryComponent.java)
- [SalaryCalculator.java](/D:/Desktop/formula/formula/service/SalaryCalculator.java)

可借鉴点：

- 将“一个计算单元”抽象成标准接口。
- 支持统一装配、统一排序、统一执行。
- 上一个组件的结果进入下一个组件的共享上下文。

这部分适合升级为平台中的“节点执行模型”。

## 3. 模式判断：责任链不是公式引擎主骨架

### 3.1 责任链模式适合什么

责任链模式适合：

- 请求按固定流程依次经过多个处理器
- 每个处理器只关心自己这一段职责
- 某些步骤可提前返回，也可继续传递

因此，责任链更适合放在“流程处理”类能力中。

### 3.2 责任链模式不适合什么

公式求值不适合以责任链作为主骨架。原因是：

- 公式依赖关系不是线性的
- 同一公式可能依赖多个上游变量
- 多个派生项之间可能形成图状依赖

因此，公式求值更适合采用：

- DAG（有向无环图）
- 拓扑排序
- 递归依赖解析

### 3.3 平台中的正确搭配

建议采用“两种模式并存”的架构：

- 责任链：处理接入、变量解析、校验、落账前检查
- DAG：处理公式、派生变量、费用依赖求值

## 4. 企业级核算平台推荐分层

建议将平台拆成六层。

### 4.1 接入层

职责：

- 处理第三方接口配置
- 处理请求方法、Header、Query、Body
- 处理鉴权
- 处理分页与特殊适配器
- 输出统一原始载荷

当前可参考落位：

- [CostVariableServiceImpl.java](/D:/Desktop/cost_platform/cost_platform_back/cost_platform/src/main/java/com/ruoyi/system/service/impl/cost/CostVariableServiceImpl.java)

建议新增能力：

- `RemoteAccessPipeline`
- `RemoteRequestBuilder`
- `RemoteAuthHandler`
- `RemoteResponseExtractor`
- `RemotePageStrategy`
- `RemoteAdapterRegistry`

### 4.2 标准对象层

职责：

- 将第三方原始数据转换成标准计费对象
- 将不同来源的数据统一成平台内部可运行的对象模型
- 接入中心字段映射优先绑定到平台变量编码；上下文路径只作为标准 JSON 直传与旧版路径模型的兼容能力

建议标准对象结构：

- `objectDimension`
- `objectCode`
- `objectName`
- `billMonth`
- `bizNo`
- `inputVars`

当前可参考落位：

- [CostRunServiceImpl.java](/D:/Desktop/cost_platform/cost_platform_back/cost_platform/src/main/java/com/ruoyi/system/service/impl/cost/CostRunServiceImpl.java)

### 4.3 变量解析层

职责：

- 统一解析平台变量
- 负责 `INPUT / DICT / REMOTE / FORMULA`
- 处理默认值、快照兜底与变量状态控制

建议使用责任链：

- `InputVariableResolver`
- `DictVariableResolver`
- `RemoteVariableResolver`
- `FormulaVariableResolver`
- `FallbackVariableResolver`

建议总入口：

- `VariableResolverChain`

### 4.4 公式与依赖引擎层

职责：

- 编译公式
- 分析依赖
- 检测环依赖
- 按依赖顺序求值
- 调用平台函数注册中心

建议模块：

- `FormulaCompiler`
- `FormulaDependencyGraph`
- `FormulaCycleDetector`
- `FormulaExecutionEngine`
- `FunctionRegistry`

这一层不建议使用责任链，应采用 DAG/依赖图。

### 4.5 费用计算层

职责：

- 费用项执行
- 单价、数量、金额计算
- 费用项之间依赖处理
- 中间结果缓存

建议模块：

- `CostNodeExecutor`
- `FeeCalculationNode`
- `QuantityResolver`
- `PriceResolver`
- `AmountResolver`

建议一个费用项对应一个节点，而不是全塞到一个大 Service 里。

### 4.6 落账与追溯层

职责：

- 结果台账
- 追溯解释
- 幂等控制
- 审计信息补齐
- 重算与覆盖策略

当前可参考落位：

- [CostRunServiceImpl.java](/D:/Desktop/cost_platform/cost_platform_back/cost_platform/src/main/java/com/ruoyi/system/service/impl/cost/CostRunServiceImpl.java)
- [index.vue](/D:/Desktop/cost_platform/cost_platform_front/src/views/cost/result/index.vue)

建议使用责任链：

- `ResultIdempotentCheckHandler`
- `ResultPrecisionHandler`
- `ResultAuditHandler`
- `ResultLedgerPersistHandler`

## 5. 平台统一命名空间设计

参考项目使用 `C` 和 `G` 两类前缀。  
企业级核算平台建议升级成更清晰的多命名空间体系：

- `I.xxx`：业务输入变量
- `D.xxx`：字典变量
- `R.xxx`：第三方远程变量
- `F.xxx`：派生公式变量
- `O.xxx`：标准对象字段
- `G.xxx`：费用结果或中间结果

这样可以解决几个长期问题：

- 变量来源不清
- 发布校验难做
- 调试日志可读性差
- 追溯解释中无法快速区分输入与派生值

## 6. 责任链在平台中的具体落位

### 6.1 第三方接口处理链

建议责任链顺序：

1. 请求参数构造
2. 鉴权处理
3. 请求发送
4. 状态码校验
5. 响应提取
6. 数据映射
7. 分页适配
8. 失败兜底

建议总类：

- `RemoteAccessPipeline`

### 6.2 变量解析责任链

建议责任链顺序：

1. 输入变量解析
2. 字典变量解析
3. 第三方变量解析
4. 派生变量求值
5. 默认值/快照兜底

建议总类：

- `VariableResolverChain`

### 6.3 发布校验责任链

建议责任链顺序：

1. 场景基础校验
2. 费用完整性校验
3. 变量完整性校验
4. 公式合法性校验
5. 依赖环检测
6. 规则引用完整性校验
7. 可发布性判断

建议总类：

- `PublishValidationChain`

### 6.4 结果落账前责任链

建议责任链顺序：

1. 幂等校验
2. 精度处理
3. 状态处理
4. 审计补齐
5. 台账落库

建议总类：

- `ResultPersistChain`

## 8. 当前平台代码的建议落位

### 8.1 已经具备雏形的部分

- 第三方变量六层配置：已初步具备接入层雏形
- 标准计费对象：已具备标准对象层雏形
- 发布治理与结果台账：已具备落账与追溯层雏形
- 运行任务与分片：已具备执行调度底座

### 8.2 需要下一步抽象的部分

- 将 `REMOTE` 相关能力从变量 Service 提炼成独立管线
- 将公式/派生变量求值从运行大 Service 中拆成引擎模块
- 将发布校验从散点逻辑收敛成校验链
- 将费用项执行从过程式逻辑收敛成节点执行器

## 9. 当前阶段推荐实施顺序

### 第一阶段

先做公式与变量引擎底座：

1. `FunctionRegistry`
2. `FormulaDependencyGraph`
3. `FormulaCycleDetector`
4. `FormulaExecutionEngine`

### 第二阶段

再做变量责任链：

1. `VariableResolverChain`
2. 输入、字典、远程、派生、兜底五类解析器

### 第三阶段

再做第三方接口平台化：

1. `RemoteAccessPipeline`
2. 请求定义、鉴权定义、响应提取、字段映射、分页策略、适配器

### 第四阶段

最后做发布校验链与费用节点执行器：

1. `PublishValidationChain`
2. `CostNodeExecutor`

## 10. 对 0.1.0 版本的定位建议

当前 `cost/dev` 阶段代码更适合定义为企业级核算平台的 `0.1.0`，其定位不是“最终形态完成”，而是：

- 核算主链可运行
- 场景、费用、变量、规则、发布、运行、结果链路已成型
- 第三方变量接入能力已具备平台化雏形
- 初始化脚本、压测、治理、调度均已有基础能力

因此，`0.1.0` 的发布建议为：

- 将 `cost/dev` 直接快进到 `main`
- 保留历史提交记录
- 打 `v0.1.0` 标签

不建议 squash 原因：

- 当前阶段提交已经按能力块累计沉淀，具备回溯价值
- 后续需要追查“某条治理能力、某条调度能力、某条接入能力”是何时进入主线时，保留历史更稳
- 企业级平台后续还需要从这些阶段性提交中拆演进路径
