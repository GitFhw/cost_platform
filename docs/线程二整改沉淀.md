# 线程二整改沉淀

更新时间：2026-03-31
工作区：`D:/Desktop/cost_platform`

## 一、线程边界重申

线程二只处理以下范围：

- 费用中心
- 变量中心
- 第三方系统接入变量

不提前实现规则中心、阶梯、发布中心、运行链的完整闭环，但可以为后续线程预留合理结构。

## 二、文档对齐后的核心要求

基于详细设计、页面原型和高级核算引擎设计，线程二至少要满足：

- 费用中心要作为场景下费用主数据的稳定承接层。
- 变量中心要支持输入、字典、接口、第三方、公式等多种来源。
- 变量配置必须服务后续规则、阶梯、发布和运行链。
- 第三方变量不能只停留在演示入口，至少要把配置模型设计清楚。
- 页面录入方式要由变量元数据驱动，不能只靠硬编码表单拼接。

## 三、当前已完成部分

- 费用中心主链已具备：查询、统计、增删改、治理预检、导出，且后端已补场景状态、费用状态、计价单位合法性硬校验。
- 变量中心主链已具备：查询、统计、增删改、治理预检、详情抽屉、第三方接入正式配置表单。
- 变量中心已补齐导入模板下载、导入预览、校验报告、执行导入、变量复制、共享影响因素模板应用。
- 第三方接入变量已完成正式配置模型：来源系统、鉴权方式、鉴权配置、字段映射、同步方式、缓存策略、失败兜底策略。
- 线程二菜单、字典、基础表结构及整改迁移脚本已建立。

## 四、当前主要偏差

### 1. 第三方接入变量的真实执行链仍偏轻量

- 正式配置模型已经补齐，但 `测试接口 / 预览数据 / 刷新缓存` 仍是轻量演示实现。
- 尚未接入真实第三方接口调用、密钥托管、同步任务、失败重试和告警。
- 预览返回的仍是示例行，不是正式生产同步链路。

### 2. 变量中心距离“数据接入与导入中心”完整形态仍有边界差距

- 线程二变量中心已具备导入、校验报告、复制和共享模板，但“独立导入中心、导入批次台账、错误报告台账”尚未建设。
- 当前导入能力收敛在变量中心页内，符合线程二整改边界，但还不是详细设计中后续完整数据接入中心形态。

### 3. 变量建模与后续运行链衔接仍需收敛

- 当前共享模板已经开始显式体现 `contract.* / storage.* / zx.* / sg.*` 命名空间思路。
- 但线程五运行输入模板和真实上下文装配链还需继续对齐 `V/C/I/F/T`、路径变量和命名空间口径。

## 五、本轮整改后剩余风险

1. 第三方接入变量目前是“配置模型正确、执行链轻量”的状态，不能直接视为生产级接入闭环。
2. 共享影响因素模板当前为内置模板库，适合线程二复用治理，但不等同于正式模板管理台。
3. 导入能力已经落在变量中心，但导入批次台账、错误报告中心、幂等批次治理属于后续数据接入中心范围。

## 六、后续建议顺序

1. 先逐条对照详细设计和原型，把费用中心、变量中心页面能力补齐。
2. 继续把第三方接入变量从“配置模型已正规化”推进到“真实执行链可托管”。
3. 统一变量命名、来源、路径、字典和映射口径，为线程三到线程五铺路。
4. 线程五整改时，要直接复用线程二沉淀出的命名空间和共享模板口径，避免再回到扁平硬编码输入。

## 七、当前建议的整改原则

- 以新项目详细设计和原型为主，不沿用老项目页面表达。
- 只在“第三方变量映射思想”和“命名空间上下文口径”不明确时，参考 `charge` 的成熟思路。
- 所有新增整改都要先过文档对照，再写代码。
