# charge 对比后的必要补齐清单

## 1. 判断口径

本清单不是简单罗列 `D:\Desktop\charge` 里“有而新项目没有”的所有能力，而是按以下口径筛选：

- 以 [`企业级核算平台详细设计.md`](/D:/Desktop/cost_platform/企业级核算平台详细设计.md) 和 [`企业级核算平台页面原型设计.md`](/D:/Desktop/cost_platform/企业级核算平台页面原型设计.md) 为主。
- 再参考 `charge` 已验证过的产品交互、治理思路和成熟度清单。
- 只保留“当前项目确实还没完善，且补上后能明显提升业务可用性/治理闭环”的项。
- 不直接照搬 `charge` 的租户模型、老表达式口径和页面结构。

## 2. 首发发布与试算的判断

### 2.1 判断结论

首发发布建议关联试算中心，但应为“警告项”，不应直接阻断发布。

### 2.2 推荐口径

- 当场景从未发布过，且最近没有成功试算记录时：
  - 在发布前检查中增加 `WARN`
  - 文案建议为“当前场景为首次发布，且尚无成功试算记录，建议先完成至少 1 次试算核对再发布”
- 当场景已有成功试算记录：
  - 通过该警告项，不再提示
- 不建议设为 `BLOCK`
  - 因为部分场景可能是演示、初始化或配置准备阶段
  - 业务方应被提醒，但不应被流程硬拦住

### 2.3 后续落点

- 模块：发布中心 + 试算中心
- 优先级：`P0`
- 建议归属：线程四/线程五联动整改

## 3. 对比后确认“必须补”的能力

### 3.1 P0：优先补齐

#### 3.1.1 首发试算提醒接入发布前检查

- `charge` 参考：
  - 运行链和试算工作台是产品主入口，发布不是孤立动作
- 当前项目现状：
  - 发布前检查已做费用/变量/规则/阶梯完整性校验
  - 但还没有把“是否做过试算”纳入提醒口径
- 必要性：
  - 高
  - 这是业务真正上线前最容易漏掉的动作

#### 3.1.2 批量试算

- `charge` 参考：
  - [`SimulationView.vue`](/D:/Desktop/charge/cost-accounting-platform/webapp/src/views/SimulationView.vue) 已同时支持单笔试算、批量试算、批量正式核算、导入预校验
- 当前项目现状：
  - `docs/风险清单` 已记录“批量试算缺失”
- 必要性：
  - 高
  - 详细设计已明确包含批量试算，且这是规则上线前的重要业务验证入口

#### 3.1.3 价格策略类型化表单继续收口

- `charge` 参考：
  - 策略配置在交互上更偏“类型化配置”，而不是让业务直接看 JSON
- 当前项目现状：
  - 规则中心主链已成型，但复杂规则/阶梯仍有偏技术化的表达
- 必要性：
  - 高
  - 这直接影响规则配置效率和业务理解成本

### 3.2 P1：建议尽快补齐

#### 3.2.1 独立发布审计页

- `charge` 参考：
  - [`PublishAuditView.vue`](/D:/Desktop/charge/cost-accounting-platform/webapp/src/views/PublishAuditView.vue)
- 当前项目现状：
  - 发布中心内已有版本台账、版本详情、差异和校验
  - 但还没有独立“发布审计查询”入口
- 必要性：
  - 中高
  - 当前规模还能在发布中心工作区承接，后续版本数增多后，独立审计页会更稳

#### 3.2.2 公式实验室工作台配置持久化

- `charge` 参考：
  - 公式实验室更偏工作台式长期维护
- 当前项目现状：
  - 已支持点选生成公式
  - 但“点选过程结构”还没有持久化
- 必要性：
  - 中高
  - 不补这层，二次编辑还会退回专家模式

#### 3.2.3 公式模板库与独立版本管理

- `charge` 参考：
  - 公式模板与模板引用关系有单独沉淀
- 当前项目现状：
  - 已有公式资产和模板雏形
  - 仍缺公式版本台账、模板库治理和历史回看
- 必要性：
  - 中高
  - 线程七后半段和线程八必须补

#### 3.2.4 全链强制按 `formulaCode` 执行

- `charge` 参考：
  - 老项目本身也存在直接表达式残留，因此这里只吸收“资产化方向”，不吸收双轨长期共存
- 当前项目现状：
  - 变量、规则、运行链已接入公式编码优先
  - 但仍保留原始表达式兜底
- 必要性：
  - 中高
  - 不强制收口，后续快照、审计、追溯口径会继续分叉

### 3.3 P2：产品可用性增强

#### 3.3.1 结构化执行时间线继续做深

- `charge` 参考：
  - 试算页对结果、变量、策略、缺失输入、公式失败等回看更完整
- 当前项目现状：
  - 已有时间线雏形
  - 但还可以继续增强结果回放、失败分组、解释导出
- 必要性：
  - 中

#### 3.3.2 模板能力平台化

- `charge` 参考：
  - 变量模板、公式模板的产品化程度更高
- 当前项目现状：
  - 共享影响因素模板还主要是内置模板库
- 必要性：
  - 中
  - 适合后续治理增强时再补成模板中心

#### 3.3.3 数据接入层产品化

- `charge` 参考：
  - 输入准备、导入预校验和批量执行衔接更自然
- 当前项目现状：
  - 第三方接入变量仍偏轻量
  - 真实字段映射、同步任务、失败重试和通知闭环还未完成
- 必要性：
  - 中

## 4. 对比后确认“不必直接照搬”的内容

以下能力在 `charge` 中存在，但当前新项目不建议直接照搬：

- 多租户/项目隔离模型
  - 当前新项目是“场景治理 + 业务域治理”口径，不必回退到老项目的租户思路
- 旧表达式兼容口径下的直接 `formulaExpr` 长期共存
  - 新项目应继续向 `formulaCode` 资产化收口
- 以顶层场景切换承载所有审计/试算入口的页面结构
  - 当前新项目已形成更清晰的中心化页面拆分，不必回退
