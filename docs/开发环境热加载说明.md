# 开发环境热加载说明

## 结论

当前项目已经具备低成本热加载能力：

- 前端：`cost_platform_front` 使用 Vite，执行 `npm run dev` 后默认支持页面和组件 HMR。
- 后端：`cost_platform_back/cost_admin` 已引入 `spring-boot-devtools`，且 `application.yml` 中已开启 `spring.devtools.restart.enabled=true`。

后端这里的能力是 DevTools 自动重启，不是 JVM 原生 HotSwap。截图中的“add method not implemented”表示当前 JVM 不能在不重启的情况下新增方法，这是 Java HotSwap 的原生限制，不是项目配置缺失。

## 推荐启动方式

前端：

```bash
cd cost_platform_front
npm run dev
```

后端：

```bash
cd cost_platform_back
mvn -pl cost_admin -am spring-boot:run
```

如果使用 IntelliJ IDEA 启动后端：

1. 使用 `RuoYiApplication` 启动应用。
2. 修改 Java 代码后执行 `Build Project` 或开启自动编译。
3. DevTools 监测到 `target/classes` 变化后会重启 Spring Boot 上下文。
4. 日志中出现 restart 相关输出时，说明 DevTools 生效。

## 能热加载的内容

- 前端 Vue 页面、组件、样式：Vite HMR，通常不刷新整页。
- 后端 Java 类、Controller、Service、Mapper 相关改动：编译后触发 DevTools 重启。
- `application.yml`、MyBatis XML、静态资源等：资源变更后触发 DevTools 重启或重新加载。

## 不能指望原生 HotSwap 处理的内容

- 新增/删除方法。
- 新增/删除字段。
- 修改类继承结构。
- 修改注解后需要 Spring 重新扫描的场景。

这些改动要依赖 DevTools 重启。若要做到不重启 JVM 的结构级热替换，需要引入 JRebel 或 DCEVM/HotswapAgent，这类方案会增加团队环境复杂度，当前阶段不建议作为默认开发依赖。

## 当前项目约定

- 保留 DevTools 作为默认开发热加载方案。
- 前端继续使用 Vite HMR。
- 提交生产包前仍以 `mvn -q -pl cost_admin -am -DskipTests package` 和 `npm run build:prod` 做验证，不依赖热加载结果判断可发布性。
