# 公式实验室业务化改造方案

更新时间：2026-04-21

## 1. 文档定位

本方案用于定义 `公式实验室` 在 `v0.2.x` 阶段的产品定位、页面结构、交互模式与后续开发边界。

它不是临时页面优化说明，而是后续改造公式实验室、规则中心公式引用、公式变量维护与试算验证体验的统一设计基线。

本方案重点回答以下问题：

- 公式实验室到底服务谁。
- 业务人员编辑的对象应该是什么。
- 中文公式与标准表达式之间如何分层。
- 资源点选、条件分支、区间档位和规则中心阶梯如何分工。
- 后续开发应该先做什么，做到什么程度才算可验收。

## 2. 当前问题判断

结合现有页面和业务人员使用习惯，当前公式实验室的核心问题不是单个控件不好用，而是产品模型还偏技术视角。

### 2.1 当前主要问题

1. 中文业务表达和标准表达式形成双编辑心智。

当前页面虽然增加了中文业务表达，但业务人员仍然需要理解 `V.SPECIAL_TEAM_HEADCOUNT`、`ROUND`、`MAX` 等标准式内容，实际还是在“看中文说明 + 维护英文公式”。

2. 资源点选行为打断编辑上下文。

每次点选资源、模板或变量时，如果页面自动切回结构编排区，会破坏业务人员的连续思路。资源点选应该是“就地插入”，而不是“切页跳转”。

3. 条件分支、区间档位和规则中心阶梯边界不清。

业务上这三者都带有“判断”和“分档”特征，但承载对象并不相同。如果边界不清，后续会出现规则中心和公式实验室重复建设、重复维护的问题。

4. 右侧长滚动工具箱不符合企业级工作台习惯。

业务人员操作企业级系统时，更习惯“主工作区 + 可收起辅助面板”，而不是固定一条很长的辅助侧栏。长滚动会让主流程被稀释，也不利于复杂资产的稳定维护。

5. 校验结果还不够面向业务。

当前更多是表达式预校验和标准式校验，业务人员真正需要的是：

- 哪个中文片段不合法。
- 哪个变量未绑定来源路径。
- 哪个区间断档或重叠。
- 当前公式为什么不能保存、不能试算、不能发布。

## 3. 产品定位

### 3.1 模块定位

公式实验室不是“技术表达式编辑器”，而是企业级核算平台中的 `可复用逻辑资产工作台`。

它负责沉淀：

- 公式资产
- 公式模板
- 公式变量逻辑
- 可跨规则复用的条件判断
- 可跨规则复用的区间映射

它不负责替代规则中心的费用治理，也不负责替代变量中心的来源映射管理。

### 3.2 主要用户

公式实验室主要服务三类角色：

1. 业务配置人员
- 默认用户。
- 只应看到中文变量名、中文函数名、中文公式结构。
- 关注公式能不能看懂、能不能校验、能不能试算、能不能复用。

2. 治理管理员
- 关注公式资产是否被引用、变更影响哪些变量和规则、是否具备版本治理能力。

3. 技术支持或高级配置人员
- 在必要时查看系统生成的标准表达式、编译日志和来源映射。
- 这部分不应成为默认操作对象。

### 3.3 设计原则

1. 业务优先：业务人员编辑中文公式，而不是维护英文编码。
2. 单主对象：主编辑对象只有一个，就是“业务公式”。
3. 系统兜底：标准表达式由系统编译生成，并用于执行与落库。
4. 就地插入：资源面板只负责插入，不打断编辑上下文。
5. 校验可定位：所有错误和警告都要能定位到具体片段。
6. 边界清晰：公式实验室沉淀可复用逻辑，规则中心维护费用规则。

## 4. 总体设计结论

公式实验室后续应采用 `业务模式主导、系统视图辅助、标准表达式只读为主` 的设计。

### 4.1 主编辑对象

业务人员主编辑对象为：`中文公式`。

例如：

- `四舍五入((21700 / 6) × 队女工人数 × (女工实际出勤 / 最大值(女工应出勤, 1)), 2)`
- `如果 货种 = 煤炭 且 班次 = 白班，那么 2，否则 1`
- `按 吞吐量 分档：0-100 => 1.2；100-300 => 1.05；其他 => 0.95`

### 4.2 系统生成对象

系统生成并维护：`标准表达式`。

例如：

- `round((21700 / 6) * V.FEMALE_TEAM_HEADCOUNT * (V.FEMALE_ACTUAL_ATTENDANCE / max(V.FEMALE_REQUIRED_ATTENDANCE, 1)), 2)`

标准表达式的职责是：

- 执行
- 落库
- 编译校验
- 调试取证
- 版本对比

默认情况下，业务人员不直接编辑标准表达式。

### 4.3 设计核心

产品模型从“双编辑模式”改为“单编辑 + 系统生成”：

- 业务人员只操作中文公式。
- 系统负责编译成标准表达式。
- 页面默认展示业务视图。
- 标准表达式折叠在系统视图中，只读为主。

## 5. 页面信息架构

页面改造后建议采用 `四区工作台` 结构。

### 5.1 顶部固定区：公式基本信息

承载字段：

- 所属场景
- 公式编码
- 公式名称
- 返回类型
- 资产类型
- 状态
- 业务说明

设计要求：

- 区块固定在页面顶部。
- 不与资源面板、试算面板混排。
- 保存、试算、另存模板等主操作放在顶部工具条。

### 5.2 主编辑区：业务公式工作台

这是页面主区域，应始终占主视觉中心。

主编辑区包含：

- 业务公式编辑器
- 公式类型切换
- 当前变量、函数、模板快速插入入口
- 实时校验提示

公式类型建议分为：

- 数值公式
- 条件公式
- 区间公式

设计要求：

- 默认进入业务模式。
- 编辑器直接展示中文变量名和中文函数名。
- 光标位置可插入变量、函数、模板、条件块、区间块。
- 不因资源点选切换 Tab、切换区域或滚动跳位。

### 5.3 下方校验区：校验与试算

承载能力：

- 语法校验
- 类型校验
- 来源路径校验
- 业务规则校验
- 试算输入
- 试算结果
- 问题定位

设计要求：

- 作为独立区块存在，不和主编辑区混杂。
- 试算输入默认支持按变量来源路径生成示例 JSON。
- 试算结果支持显示结果值、变量取值、命中路径和错误原因。

### 5.4 底部折叠区：系统视图

承载内容：

- 标准表达式
- 编译结果
- 变量映射明细
- 引用关系
- 生成建议
- 调试日志

设计要求：

- 默认折叠。
- 普通业务人员默认只读。
- 高级用户可展开查看，但不应成为主操作入口。

## 6. 资源点选交互方案

### 6.1 原则

资源点选必须改成 `就地插入`，不应再以“切换 Tab -> 点选 -> 跳回编排区”的方式打断用户。

### 6.2 建议交互

页面提供一个统一的 `插入资源` 入口，点击后从右侧打开资源抽屉。

抽屉中分为五类：

- 变量
- 常用函数
- 条件模板
- 区间模板
- 已有公式片段或模板

每一项都支持：

- 搜索
- 最近使用
- 中文名显示
- 编码辅助查看
- 一键插入到当前光标位置

### 6.3 插入行为

插入时遵循以下规则：

1. 不切页面。
2. 不切 Tab。
3. 不重置光标位置。
4. 插入后保持编辑器焦点。
5. 如果插入的是结构块，自动补齐占位符。

例如：

- 插入变量：`队女工人数`
- 插入函数：`四舍五入( , 2)`
- 插入条件块：`如果 [条件]，那么 [结果]，否则 [结果]`
- 插入区间块：`按 [变量] 分档：` 并自动生成首个档位

## 7. 中文公式语法设计

### 7.1 设计目标

中文公式不是“说明文案”，而是业务人员真正编辑的可执行业务公式。

系统应支持把中文公式解析成内部 AST，再编译成标准表达式。

### 7.2 建议支持的基础语法

#### 7.2.1 基础运算

支持：

- `+`
- `-`
- `×`
- `÷`
- `(` `)`

#### 7.2.2 中文函数

第一阶段建议支持：

- `四舍五入(值, 位数)` -> `round()`
- `最大值(a, b)` -> `max()`
- `最小值(a, b)` -> `min()`
- `绝对值(a)` -> `abs()`
- `取整(a)` -> `floor()` 或平台约定函数
- `安全除法(分子, 分母, 兜底值)` -> 平台安全函数

#### 7.2.3 中文条件

支持结构：

- `如果 条件，那么 结果，否则 结果`
- `如果 条件1，那么 结果1；如果 条件2，那么 结果2；否则 结果3`

#### 7.2.4 中文区间

支持结构：

- `按 [变量] 分档：0-10 => 1；10-20 => 2；其他 => 0`

#### 7.2.5 中文变量

业务人员看到并编辑的是中文变量名，例如：

- `队女工人数`
- `女工实际出勤`
- `女工应出勤`

系统内部通过变量唯一标识映射到真实变量编码，再编译为标准表达式。

### 7.3 中文公式与标准表达式映射规则

#### 7.3.1 变量层

- 页面展示：中文变量名
- 系统编译：真实变量编码，例如 `V.FEMALE_TEAM_HEADCOUNT`
- 运行取值：再根据变量中心配置的 `dataPath` 读取输入 JSON

#### 7.3.2 函数层

- 页面展示：中文函数名
- 系统编译：平台标准函数

#### 7.3.3 条件和区间层

- 页面展示：中文条件结构和区间结构
- 系统编译：条件表达式或区间映射表达式

## 8. 校验设计

公式实验室必须从“表达式能不能跑”升级为“业务人员能不能知道哪里有问题”。

建议校验分五层。

### 8.1 语法校验

检查：

- 括号是否匹配
- 中文函数参数是否完整
- 条件块结构是否完整
- 区间配置是否完整

输出要求：

- 指向具体片段
- 提示错误位置和修复建议

### 8.2 类型校验

检查：

- 数值变量是否被当作字典值比较
- 字典变量是否被拿来做数值运算
- 返回类型是否与公式结果一致

### 8.3 来源映射校验

检查：

- 引用变量是否存在
- 变量是否绑定来源路径
- 变量来源类型是否和使用方式一致

### 8.4 业务校验

检查：

- 是否存在除数为 0 风险
- 条件是否缺失 `否则`
- 区间是否断档或重叠
- 兜底结果是否缺失

### 8.5 治理校验

检查：

- 当前公式是否被变量中心引用
- 当前公式是否被规则中心引用
- 修改后会影响哪些费用、规则、发布版本

### 8.6 校验结果展示要求

每一条校验结果必须包含：

- 级别：错误 / 警告 / 提示
- 发生位置：具体中文片段或区块
- 问题说明
- 修复建议
- 是否阻断保存
- 是否阻断试算
- 是否阻断发布

## 9. 与规则中心的边界

### 9.1 公式实验室保留能力

公式实验室负责：

- 可复用逻辑资产
- 公式模板
- 公式变量计算逻辑
- 跨规则复用的条件判断
- 跨规则复用的区间映射

### 9.2 规则中心保留能力

规则中心负责：

- 单费用规则维护
- 规则条件维护
- 计量变量绑定
- 单费用阶梯费率治理
- 发布校验、差异分析和治理检查

### 9.3 边界原则

如果一段分档逻辑只服务于 `一个费用规则`，优先留在规则中心阶梯。

如果一段条件或分档逻辑需要被 `多个规则或多个变量复用`，再沉淀为公式资产进入公式实验室。

一句话总结：

- 公式实验室管“逻辑资产”。
- 规则中心管“费用治理”。

## 10. 权限与视图策略

建议引入两层视图。

### 10.1 业务视图

默认视图，面向业务人员。

只开放：

- 中文公式编辑
- 资源插入
- 校验
- 试算
- 模板保存

### 10.2 系统视图

折叠区或高级视图，面向治理管理员和技术支持。

开放：

- 标准表达式只读查看
- 编译结果查看
- 变量映射查看
- 系统诊断日志查看

原则上不建议把“直接编辑标准表达式”作为默认能力开放给业务人员。

## 11. 页面原型建议

建议页面大致结构如下：

```text
[公式基本信息]
场景 / 编码 / 名称 / 返回类型 / 状态 / 保存 / 试算 / 另存模板

[业务公式工作台]
公式类型切换：数值公式 | 条件公式 | 区间公式
中文公式编辑器
校验提示（就地）
插入资源按钮 -> 右侧抽屉

[校验与试算]
语法校验 | 类型校验 | 来源映射 | 业务校验
试算 JSON | 结果值 | 变量取值 | 命中说明

[系统视图（折叠）]
标准表达式
编译结果
变量映射
引用影响
```

## 12. 老 charge 项目可吸收点

老 `charge` 项目在公式工作台上有两点可以吸收，但不建议原样搬迁：

1. 可吸收点
- 资源、模板、函数、变量放在一个统一工具域内。
- “插入”而不是“切换”的心智是对的。
- 试算上下文和结果调试离编辑区较近，调试链路直观。

2. 不建议直接继承的部分
- 计算器式按钮区偏技术，不适合现在这套治理主界面。
- 英文表达式仍是主编辑对象，不符合当前业务化方向。
- 侧栏过长，容易把主流程冲散。

因此，新的公式实验室应吸收其“工作台 + 插入工具 + 试算联动”的长处，但改造成 `中文业务公式优先` 的企业级核算治理界面。

## 13. 开发优先级建议

### 13.1 第一阶段：先把产品模型改对

优先级：`P1`

目标：让业务人员真正只操作中文公式。

本阶段必须完成：

1. 业务公式主编辑区改成中文公式主模式。
2. 标准表达式改成系统视图，只读为主。
3. 资源点选改成右侧抽屉插入，不再自动跳转结构编排。
4. 条件分支、区间档位和规则阶梯边界说明内置到页面。
5. 校验结果至少支持“定位到中文片段”。

### 13.2 第二阶段：把体验做顺

优先级：`P1`

本阶段建议完成：

1. 中文函数库和条件块模板。
2. 最近使用资源、常用模板置顶。
3. 试算结果按中文变量名解释。
4. 修改公式后显示影响范围。
5. 模板与公式资产的统一保存入口。

### 13.3 第三阶段：增强治理能力

优先级：`P2`

本阶段建议完成：

1. 公式版本对比。
2. 公式发布影响分析。
3. 公式引用拓扑查看。
4. 公式变更审计。
5. 标准表达式编译日志明细。

## 14. 验收口径

公式实验室业务化改造完成后，应满足以下验收标准：

1. 业务人员不需要直接编辑 `V.变量编码` 就可以完成公式维护。
2. 资源点选不会导致页面自动切换主 Tab 或丢失当前编辑上下文。
3. 中文公式可自动编译为标准表达式，并可执行试算。
4. 校验结果能定位到具体中文公式片段。
5. 业务人员能分清“公式实验室逻辑资产”和“规则中心费用治理”的边界。
6. 高级用户仍可查看标准表达式和映射关系，但不影响业务主流程。

## 15. 本方案对应的产品决策

基于当前阶段和企业级产品定位，确认以下决策：

1. 公式实验室后续主编辑对象为 `中文公式`，不是英文标准表达式。
2. 标准表达式作为系统生成产物存在，默认只读，不再作为业务主入口。
3. 资源点选采用 `抽屉式就地插入`，不采用“切 Tab 再返回”的方式。
4. 条件分支和区间档位继续保留，但定位为 `逻辑资产编排能力`，不替代规则中心阶梯治理。
5. 后续所有公式实验室改造，都应以本方案为开发与验收基线。
