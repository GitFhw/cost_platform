# v0.2.0 开发执行方案

## 1. 执行原则

`v0.2.0` 必须严格按照以下文件落实开发，不再按零散口头结论推进：

- [企业级核算平台底层设计与责任链落位方案.md](/D:/Desktop/cost_platform/docs/企业级核算平台底层设计与责任链落位方案.md)
- [v0.1.0版本说明.md](/D:/Desktop/cost_platform/docs/v0.1.0版本说明.md)

本文件用于把设计基线转成可执行开发约束。

## 2. 总体目标

`v0.2.0` 的目标不是继续平铺页面功能，而是把 `v0.1.0` 的平台雏形提升为“更稳定的企业级核算底座”。

重点不是新增业务口径，而是把下面几层真正收成平台能力：

- 第三方接入层
- 标准对象层
- 变量解析层
- 公式与依赖引擎层
- 费用计算层
- 发布校验层

同时明确一条 0.2.0 重构原则：

- 如果 `v0.1.0` 的内部实现已经不适合作为长期底座，`v0.2.0` 可以直接删除旧实现并切换到新的平台化设计。
- `v0.2.0` 追求的是更好的底层方案，不以兼容旧代码结构为目标，不为了保留旧写法持续叠加补丁。
- 需要保留的是业务口径、关键数据结果和追溯能力，不是旧版内部实现本身。

## 3. 严格执行约束

### 3.1 不允许继续的开发方式

`v0.2.0` 阶段，不允许继续采用以下方式：

- 直接把新逻辑塞进单个大 Service
- 新增第三方接口时按系统逐个硬编码
- 派生变量和公式依赖继续只靠人工约定
- 发布前完整性校验继续散落在不同方法里
- 费用项执行逻辑继续与接入、取数、落账强耦合
- 明知新设计已经确定，仍为了兼容 `v0.1.0` 旧实现而继续叠加过渡补丁
- 把“先兼容旧逻辑，后续再重构”当成默认策略，继续积累结构债务

### 3.2 必须遵守的开发方式

`v0.2.0` 阶段，新增能力必须优先落到下面这些抽象层里：

- 第三方接口能力先考虑进入 `RemoteAccessPipeline`
- 变量来源处理先考虑进入 `VariableResolverChain`
- 公式求值能力先考虑进入 `FormulaExecutionEngine`
- 函数扩展先考虑进入 `FunctionRegistry`
- 发布检查先考虑进入 `PublishValidationChain`
- 费用执行先考虑进入 `CostNodeExecutor`

如果某项能力无法落入这些抽象层，必须先说明原因，再决定是否临时旁路。

## 4. v0.2.0 开发主线

在进入各主线前，统一补充一条实现约束：

- `v0.2.0` 默认允许直接替换 `v0.1.0` 中不合理的内部实现。
- 已被新抽象接管的旧方法、旧分支、旧中间层，应在验证通过后及时删除。
- 不保留长期不会再使用的双轨逻辑；如果暂时保留，必须在版本清单中明确标记为待删除过渡层。

### 4.1 主线一：公式与依赖引擎平台化

目标：

- 把派生变量与公式求值从过程式代码中抽出来
- 建立统一依赖图
- 建立循环依赖检测
- 建立统一函数注册中心

必须产出：

- `FunctionRegistry`
- `FormulaCompiler`
- `FormulaDependencyGraph`
- `FormulaCycleDetector`
- `FormulaExecutionEngine`

验收标准：

- 派生变量保存时可检测非法引用
- 发布前可检测环依赖
- 运行时不再依赖手工顺序约束

### 4.2 主线二：变量解析责任链

目标：

- 把 `INPUT / DICT / REMOTE / FORMULA / FALLBACK` 收成统一变量解析责任链

必须产出：

- `VariableResolverChain`
- `InputVariableResolver`
- `DictVariableResolver`
- `RemoteVariableResolver`
- `FormulaVariableResolver`
- `FallbackVariableResolver`

验收标准：

- 变量来源新增不再需要继续堆 if/else
- 变量测试、预览、运行口径一致

### 4.3 主线三：第三方接入层平台化

目标：

- 把当前第三方变量六层配置从“变量附属能力”升级成“平台接入层能力”

必须产出：

- `RemoteAccessPipeline`
- `RemoteRequestBuilder`
- `RemoteAuthHandler`
- `RemoteResponseExtractor`
- `RemoteFieldMapper`
- `RemotePageStrategy`
- `RemoteAdapterRegistry`

验收标准：

- 接一个新的第三方接口时，优先通过配置完成
- 只有极特殊差异才允许补适配器

### 4.4 主线四：发布校验链

目标：

- 把发布前检查从散点逻辑提升为平台标准校验链

必须产出：

- `PublishValidationChain`
- 场景、费用、变量、公式、规则、依赖、可发布性等校验器

验收标准：

- 发布失败时能明确指出是哪一层未通过
- 发布检查结果可回看、可追溯

### 4.5 主线五：费用节点执行器

目标：

- 把费用执行能力从当前运行大 Service 中逐步抽象成节点执行模型

必须产出：

- `CostNodeExecutor`
- 费用项节点
- 数量节点
- 单价节点
- 金额节点

验收标准：

- 费用项执行口径更清晰
- 费用依赖关系可表达

## 5. 建议实施顺序

`v0.2.0` 建议严格按以下顺序实施：

1. 公式与依赖引擎平台化
2. 变量解析责任链
3. 第三方接入层平台化
4. 发布校验链
5. 费用节点执行器

原因：

- 公式与依赖是平台内核
- 变量解析是内核入口
- 第三方接入是扩展入口
- 发布校验是治理能力
- 费用节点执行器是最终执行抽象

## 6. 开发落地要求

### 6.1 文档要求

每条主线开工前，都要补齐：

- 设计目标
- 影响范围
- 新增核心类
- 与现有类的迁移关系

### 6.2 代码要求

每条主线收尾时，都至少要有：

- 代码实现
- 自动化验证
- 页面或接口联调结果
- 文档回写

### 6.3 提交要求

`v0.2.0` 阶段提交要按能力块提交，不再混杂过多平行主题。  
建议提交粒度：

- 一条主线一个或多个连续提交
- 每个提交具备清晰主题
- 不在同一提交里混放无关治理、页面、性能、接入改动

## 7. 命名空间约束

`v0.2.0` 开始，新增公式、派生变量与内部执行上下文应逐步统一到以下命名域：

- `I.xxx`：输入变量
- `D.xxx`：字典变量
- `R.xxx`：远程变量
- `F.xxx`：派生变量
- `O.xxx`：对象字段
- `G.xxx`：费用结果

旧口径可兼容，但新能力优先按新命名空间设计。

## 8. 版本完成判定

`v0.2.0` 只有在以下条件同时满足时，才可定义为阶段完成：

- 公式与依赖引擎已抽象完成
- 变量解析责任链已抽象完成
- 第三方接入层已平台化
- 发布校验链已形成统一入口
- 费用执行抽象已初步落位
- 对应文档、测试、回归结果已同步完成
- 已完成至少一轮与 `v0.1.0` 同口径的压测对比
- 已完成至少一轮与 `v0.1.0` 的变量/规则结果对比回归，并明确记录差异

只完成页面增强或零散接口补丁，不视为 `v0.2.0` 完成。

压测与结果对比要求：

- `v0.2.0` 完成后，必须按 `v0.1.0` 已通过的压测口径至少重跑一轮。
- 压测结果不仅看“是否能跑通”，还要对比 `吞吐 / 总耗时 / 失败分片数 / 稳定性`。
- 变量与规则改造后，不默认认为结果一定与 `v0.1.0` 完全一致；必须通过对比回归确认。
- 如果结果存在差异，必须区分：
  - 旧实现缺陷被修正导致的合理差异
  - 新设计引入的非预期行为差异
- 所有差异都必须形成书面记录，才能决定是否放行。

## 9. 结论

`v0.2.0` 不再是“继续补功能”，而是“严格按设计基线把平台底层能力做实”。  
后续所有 0.2.0 范围内开发，都应以本文件作为执行约束文件。
