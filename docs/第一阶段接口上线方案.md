# 第一阶段接口上线方案

## 1. 上线目标

- 第一阶段不要求前台页面全部闭环，只要求“费用维护完成后，第三方系统可以通过后端接口拿到输入格式并完成费用计算”。
- 第三方系统负责把业务明细整理成我方费用规则需要的字段结构，再调用我方接口取费。
- 费用结果若需要留痕、回看、追溯，则走正式核算任务接口；若只需要同步取价，则优先走按费用同步计算接口。

## 2. 推荐接口调用顺序

1. 调用 `GET /cost/run/input-template/fee`
2. 按返回的 `includedInTemplate=true` 字段组织第三方批量数据
3. 调用 `POST /cost/run/fee/calculate` 做同步取费
4. 若需要结果台账、追溯解释、账期沉淀，再调用 `POST /cost/run/task/submit`
5. 调用 `GET /cost/run/result/list` 和 `GET /cost/run/result/{resultId}` 做结果回查

## 3. 当前可用接口

- `GET /cost/run/input-template/fee`
  - 用途：按费用返回第三方需要准备的数据格式
  - 关键参数：`sceneId`、`versionId`、`feeId`/`feeCode`、`taskType`
- `POST /cost/run/fee/calculate`
  - 用途：按单项费用同步计算，适合第一阶段接口化接入
  - 请求体关键字段：`sceneId`、`versionId`、`feeId`/`feeCode`、`billMonth`、`inputJson`
  - `inputJson` 支持单对象和对象数组
  - 可选字段：`includeExplain=true` 时返回轻量命中解释，便于第三方在联调或排障时直接看到候选规则、命中组合组、阶梯和定价说明
- `POST /cost/run/task/submit`
  - 用途：正式核算落库，生成任务、分片、结果台账和追溯记录
- `GET /cost/run/task/list`
  - 用途：按请求号回查正式任务
  - 当前已支持 `requestNo` 精确筛选
- `GET /cost/run/result/list`
  - 用途：费用明细查询
  - 当前已支持的关键筛选：`taskId`、`requestNo`、`feeId`、`feeCode`、`objectCode`、`objectName`、`billMonth`、`resultStatus`
- `GET /cost/run/result/{resultId}`
  - 用途：查看单条费用结果及其追溯解释

## 4. 2026-04-03 联调结果

- `STORAGE_KEEP_FEE` 的费用模板接口已返回 3 个输入字段和 1 个公式派生字段，第三方只需准备 `CUSTOMER_LEVEL`、`OCCUPIED_AREA`、`BILL_DAYS`
- `POST /cost/run/fee/calculate` 已真实联调通过
  - 同一批 2 条样例数据返回 1 条 `SUCCESS`、1 条 `NO_MATCH`
  - 成功记录命中规则 `STORAGE_KEEP_RATE_A_B`
  - `includeExplain=true` 时，成功记录可直接返回条件/定价/时间线解释，未命中记录也会返回候选规则与未命中说明
- `POST /cost/run/task/submit -> GET /cost/run/task/{taskId} -> GET /cost/run/result/list -> GET /cost/run/result/{resultId}` 已真实联调通过
- `GET /cost/run/task/list?requestNo=...` 和 `GET /cost/run/result/list?requestNo=...` 已联调通过，可直接按第三方请求号回查
- 联调过程中顺手修复了异步线程回写账期、告警、审计时直接依赖登录态导致任务被误标失败的问题

## 5. 百万级数据性能结论

- 现有“按费用找规则”本身不是主要瓶颈
  - 运行快照已按 `feeCode -> rules` 建立 `Map`，按费用找规则是 O(1) 哈希命中
- 真正影响百万级吞吐的热点在这里
  - 如果一次调用要把场景下全部费用都跑一遍，单条对象的总复杂度会随费用数和每费规则数一起放大
  - 每条规则命中时仍要做条件判断、表达式执行和金额计算
  - 正式核算场景下还要写结果台账、追溯记录、任务明细和分片状态
- 当前已经做过的热点优化
  - 条件组在快照装载时预分组，不再对每条输入重复 regroup
  - 条件上下文在规则匹配时只合并一次，不再对每个条件重复构造
  - 新增 `POST /cost/run/fee/calculate`，让第一阶段第三方可以只算目标费用，避免对单条对象遍历全量费用
  - 运行快照已按费用预计算“执行变量闭包”，单费用调用时只计算该费用命中的条件、数量、公式所需变量

## 6. 第一阶段接入建议

- 第三方按费用分批调用 `POST /cost/run/fee/calculate`，不要把所有费用混在一次同步取费里
- 同步取费优先用于“实时算价/批量取价”；需要留痕时再提交正式核算任务
- `includeExplain` 建议只在联调、抽样复核或异常排查时打开，大批量正式调用默认关闭，避免回包过重
- 百万级数据建议调用方按批分页提交，不建议单次塞入超大 JSON 数组
- 结果回查优先使用 `taskId + feeCode + objectCode` 组合定位
- 真正上线前仍建议单独补一次百万级压测，重点看表达式执行耗时、结果写库耗时和调用方批次大小
