# 百万级压测方案与报告模板

## 1. 目标

本方案用于验证企业级核算平台在“标准计费对象批量装载 -> 正式核算 -> 分片执行 -> 结果落账”主链上的容量上限、吞吐能力和恢复能力。

本轮压测不把行业特例写死进平台，只验证通用能力：

- 批量输入装载是否可分页、可分片、可恢复
- 发布快照运行态是否能稳定支撑大批量任务
- 表达式执行、规则匹配、结果写库是否存在明显瓶颈
- 失败分片重试、缓存刷新、Redis 锁控制是否会在高并发下失效

## 2. 当前产品口径

当前正式核算链已经具备这些基础能力：

- 输入批次流式解析写库
- 输入批次详情服务端分页
- 正式核算按分片执行
- 分片失败重试、明细失败重试
- 任务取消
- Redis 分布式锁

当前代码里的关键默认值：

- 分片大小：`500`
- 默认并发：`8`
- 输入批次写库缓冲：`200`

## 3. 压测场景

建议至少覆盖 5 类场景：

1. 单账期单场景单版本正式核算
2. 同账期重复提交拦截与幂等验证
3. 大批量输入批次装载与正式核算联动
4. 失败分片重试与恢复统计
5. 同节点多任务并发正式核算

建议的规模梯度：

| 梯度 | 记录量 | 目的 |
|---|---:|---|
| S1 | 1,000 | 验证链路正确与基础耗时 |
| S2 | 10,000 | 验证分片、并发和写库节奏 |
| S3 | 100,000 | 验证开发/测试环境上限 |
| S4 | 500,000 | 验证接近生产规模的稳定性 |
| S5 | 1,000,000 | 形成上线放行结论 |

## 4. 指标口径

至少记录这些指标：

- 输入 JSON 构建耗时
- 输入批次创建耗时
- 正式任务提交耗时
- 任务总耗时
- 平均吞吐：`记录数 / 总耗时`
- 分片总数
- 失败分片数
- 成功数 / 失败数
- 数据库 CPU、内存、磁盘 IO
- 应用 CPU、堆内存、GC 次数
- Redis 锁等待/冲突情况

建议附加指标：

- 结果台账写入 TPS
- 追溯台账写入 TPS
- 表达式执行平均耗时
- 单条对象平均命中规则数

### 7.3 风险记录

- 装载阶段：
- 装载阶段：`100,000` 记录已在本地开发环境跑通，并且本轮已修复“任务明细初始化一次性全量写入导致超大 SQL 报文”的问题，当前装载链改为平台自控的分段写入。
- 规则匹配阶段：当前基线未出现异常放大，说明已发布快照下的规则匹配链在 `10,000` 量级仍可控。
- 表达式执行阶段：当前样本公式复杂度较低，尚不足以代表复杂公式资产场景，后续应补高复杂度公式集压测。
- 结果写库阶段：`100,000` 量级无失败分片，但总耗时已进入 `10+` 分钟区间，仍需继续观察 `500,000+` 时台账与追溯双写对数据库的持续压力；本轮已额外收口“提交任务时全量改写批次明细状态”和“任务完成时从结果台账全表汇总账期金额”两个百万级瓶颈。
- Redis/锁冲突：当前已补本地“2 任务 * 1,000 条”“4 任务 * 1,000 条”“8 任务 * 1,000 条”的同节点并发基线，以及“4 任务 * 1,000 条同账期并发”基线，说明当前任务提交锁、任务派发锁、恢复扫描协调锁和分片执行链在“同场景不同账期”与“同账期并发”口径下都可稳定并行；但仍未覆盖跨节点抢占和多节点统一调度。
- 恢复能力：已有失败分片重试和失败明细重试自动化回归，但压测报告尚未补“高载荷下故障恢复”专项结果。

### 7.4 放行结论

- 是否满足第一阶段上线：从本地 `1,000 / 10,000 / 100,000 / 500,000 / 1,000,000` 单任务基线，以及“2 任务 * 1,000 条”“4 任务 * 1,000 条”“8 任务 * 1,000 条”并发基线和“4 任务 * 1,000 条同账期并发”基线看，正式核算主链已具备企业级基本吞吐与并行执行能力；正式放行仍取决于目标环境 `500,000 / 1,000,000` 与更高并发场景结果。
- 推荐调用方批次大小：第一阶段建议外部调用方按单费用、单账期、单批次 `1,000 ~ 10,000` 条控制调用窗口，`100,000` 级及以上优先走导入批次或分页接入方案。
- 是否需要扩库/扩容：当前无法仅凭本地开发环境结论决定，需先补目标环境 `100,000 / 500,000 / 1,000,000` 实测。
- 是否需要推进分布式调度：需要，尤其是在多节点正式核算与并发任务场景下，`PERF-002` 仍应继续推进。
- 是否需要继续优化写库或表达式执行：需要，重点是 `100,000+` 量级下的批量写入、追溯写入和复杂公式执行链。

## 9. 本轮设计收口

- 本轮压测先暴露了 `100,000` 条下任务明细初始化批量 SQL 报文过大问题，根因是平台一次性全量写入 `cost_calc_task_detail`，过度依赖数据库 `max_allowed_packet`。
- 当前已改为平台自控的分段写入策略，任务明细和分片台账初始化均按固定窗口批量插入，不再把容量边界交给数据库默认参数。
- 这个调整符合企业级核算平台设计：批量链路的容量控制应由平台掌握，而不是由底层默认配置“碰运气”。
- `500,000` 量级继续暴露了第二个企业级问题：压测过程本身不能制造超大测试产物。当前已把性能基线测试日志压到 `WARN`，避免 surefire 报告因系统输出失控而反向污染压测结论。
- `1,000,000` 量级进一步暴露了第三个企业级问题：批次一旦提交，不应再用一次全表更新去改写所有输入明细状态。当前已收口为“仅更新批次头为已提交”，输入明细继续作为原始输入台账保留，避免百万级批量更新拖垮任务提交链。
- `1,000,000` 量级还验证了第四个企业级问题：任务收尾和账期汇总不应依赖结果台账明细全表求和。当前已把金额汇总上移到分片执行结果，在分片台账落 `amount_total`，任务收尾和账期刷新统一改读分片汇总，避免 finish 阶段再次扫描百万级结果台账。
- 本轮新增的并发基线说明：当前平台在“同场景、不同账期”的 2 任务、4 任务、8 任务并行执行口径，以及“同场景、同账期”的 4 任务并行执行口径下，输入批次、任务提交锁、任务派发锁、恢复扫描协调锁和分片执行链都可以稳定协同；这为后续跨节点调度和更高并发基线提供了当前阶段证据。
