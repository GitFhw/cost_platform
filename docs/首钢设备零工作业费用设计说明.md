## 4. 暂不并入自动计价的项目

### 4.1 板车托车费

- 项目主数据存在：`14108521985821494`
- 业务口径：`500 元/次`
- 当前处理方式：单独作为手工补录费项维护，不并入接口自动计价链路

这样处理的原因：

- 当前你给出的真实费率文件里未看到这一项和其他设备零工项同样的接口费率行
- 业务上它更像特殊固定收费项，不一定和普通“项目 ID + 数量”完全同构
- 先独立建费项，后续如果接口已稳定返回它的项目 ID 和数量，再并回自动链路即可

### 4.2 抓车抓高

- 项目主数据存在：`13791504124134451`
- 当前没有确认的真实费率
- 本次不纳入自动计价规则

这样处理的原因：

- 如果先把没有真实费率的项塞进规则，会造成运行时命中项目但无正确单价
- 这类不完整配置在正式核算里风险比“先不纳入”更高

## 5. 上下文设计

这次设备场景的核心上下文刻意保持最小化，避免后面为了适配接口又反复打补丁。

### 5.1 自动计价主上下文

自动计价链路只保留两个核心变量：

- `DEVICE_ODD_PROJECT_ID`
- `DEVICE_ODD_QUANTITY`

含义如下：

- `DEVICE_ODD_PROJECT_ID`：接口返回的作业项目 ID，用来命中分组定价规则
- `DEVICE_ODD_QUANTITY`：接口返回的实际计费数量，可能是小时、吨、块等

这样设计的好处：

- 接口只要能稳定返回项目 ID 和数量，就能直接驱动计价
- 费率与单位都固化在规则分组中，不需要第三方系统再传价格
- 后续新增新的设备零工项目时，只需加字典项、分组价和规则条件，不必重改计算主链路

### 5.2 手工补录上下文

为板车托车费单独保留：

- `FLATBED_TOW_COUNT`

含义如下：

- 表示板车托车费的计费次数
- 固定按 `500 元/次` 计价

这样做可以让特殊收费项和标准设备零工项解耦，后续如果接入接口，也只需要替换这条费项的来源，不会干扰设备零工主规则。

## 6. 场景与费用设计

### 6.1 场景

- 场景编码：`SHOUGANG-ORE-DEVICE-001`
- 场景名称：`首钢矿石设备费用`
- 默认对象维度：`协力单位`

当前费用项也统一按 `协力单位` 作为核算对象维度。零工作业明细仍作为计费输入上下文使用，费用最终归集到协力单位。

### 6.2 变量组

- `SG_EQ_BASE`：设备零工基础输入
- `SG_EQ_MANUAL`：设备零工手工补充

### 6.3 变量

| 变量编码 | 变量名称 | 数据类型 | 说明 |
| --- | --- | --- | --- |
| `DEVICE_ODD_PROJECT_ID` | 设备零工作业项目ID | STRING/DICT | 命中设备零工作业项目 |
| `DEVICE_ODD_QUANTITY` | 设备零工作业数量 | NUMBER | 统一作业数量 |
| `FLATBED_TOW_COUNT` | 板车托车次数 | NUMBER | 手工补录 |

### 6.4 费用项

| 费用编码 | 费用名称 | 处理方式 |
| --- | --- | --- |
| `SG_DEVICE_ODD_JOB_FEE` | 设备零工作业费 | 自动计价 |
| `SG_DEVICE_FLATBED_TOW_FEE` | 板车托车费 | 手工补录 |

## 7. 规则设计

### 7.1 设备零工作业费

规则编码：

- `SG_DEVICE_ODD_JOB_GROUPED_01`

规则设计：

- 定价类型：固定单价
- 定价模式：分组定价
- 数量变量：`DEVICE_ODD_QUANTITY`
- 分组命中字段：`DEVICE_ODD_PROJECT_ID`

每个项目 ID 对应一个分组，每个分组内维护：

- 项目标签
- 项目 ID
- 单位
- 单价

### 7.2 板车托车费

规则编码：

- `SG_DEVICE_FLATBED_TOW_RATE_01`

规则设计：

- 定价类型：固定单价
- 数量变量：`FLATBED_TOW_COUNT`
- 固定单价：`500`
- 单位：`次`

## 8. 为什么这版设计适合后续彻底跑通

这版不是“先硬塞进去跑”，而是按后续正式联调可演进的方式做的：

- 主链路最小上下文稳定：项目 ID + 数量
- 费率落在平台内部，不依赖第三方重复传价
- 特殊收费项单独拆出，避免污染主规则
- 未确认费率的项不提前硬配，避免生成假规则
- 后续接接口变量时，只需要补来源配置，不需要重做费用模型
