# 目标环境压测执行手册

## 1. 目的

本手册用于执行 [统一缺陷与开发清单](/D:/Desktop/cost_platform/docs/统一缺陷与开发清单.md) 中的 `PERF-006`：

- 在目标环境完成 `500,000 / 1,000,000` 单任务正式核算压测
- 在目标环境补高并发任务组合压测
- 在目标环境补长期稳定性压测
- 沉淀可审计的任务、分片、结果台账证据

本手册不绑定任何行业口径，只验证企业级核算平台通用能力。

## 2. 适用前提

执行前需确认：

- 已部署包含当前性能与调度能力的版本
- MySQL、Redis、应用节点已联通
- 已存在可用场景与已发布版本
- 执行账号具备：
  - 启动压测脚本权限
  - 查看应用日志权限
  - 查询数据库只读权限

建议提前记录：

- 场景编码
- 场景 ID
- 版本 ID
- 账期
- 节点数量
- 节点标识配置 `cost.dispatch.node-id`

## 3. 标准执行矩阵

建议至少执行以下矩阵：

| 类别 | 场景标签 | 参数 |
| --- | --- | --- |
| 单任务 | `single_500000` | `500,000` 条 |
| 单任务 | `single_1000000` | `1,000,000` 条 |
| 并发 | `concurrent_multi_4x10000_r1` | `4` 任务 * `10,000` 条，不同账期 |
| 并发 | `concurrent_multi_8x10000_r1` | `8` 任务 * `10,000` 条，不同账期 |
| 并发 | `concurrent_same_4x10000_r1` | `4` 任务 * `10,000` 条，同账期 |
| 长稳 | `concurrent_multi_4x10000_r1~r3` | `4` 任务 * `10,000` 条，连续 `3` 轮 |

如果目标环境资源较强，可额外增加：

- `8` 任务 * `50,000` 条
- `4` 任务 * `100,000` 条

## 4. 执行脚本

项目已提供目标环境压测脚本：

- [run-target-perf.ps1](/D:/Desktop/cost_platform/cost_platform_back/bin/run-target-perf.ps1)

默认输出：

- 压测日志：每个场景一个 `.log`
- 场景 JSON：每个场景一个 `.json`
- 汇总结果：`summary.md`

### 4.1 推荐执行方式

在仓库根目录执行：

```powershell
.\cost_platform_back\bin\run-target-perf.ps1 `
  -SceneCode "SHOUGANG-ORE-HR-001" `
  -BillMonth "2027-04" `
  -SingleRecordCounts 500000,1000000 `
  -ConcurrentMatrices "4x10000","8x10000","4x10000@same" `
  -SoakRounds 3
```

如果环境中已知 `sceneId / versionId`，优先直接传入：

```powershell
.\cost_platform_back\bin\run-target-perf.ps1 `
  -SceneId 10005 `
  -VersionId 26 `
  -BillMonth "2027-04"
```

## 5. 压测后取证

数据库取证统一使用：

- [目标环境压测取证SQL.sql](/D:/Desktop/cost_platform/docs/目标环境压测取证SQL.sql)

建议每个关键场景至少沉淀三类证据：

1. 脚本输出证据
- `summary.md`
- 对应场景 `.json`
- 对应场景 `.log`

2. 数据库证据
- 任务头信息
- 分片 owner 分布
- 结果台账汇总

3. 运行态证据
- 应用节点 CPU / 内存 / GC
- MySQL CPU / IO
- Redis 锁冲突或等待情况

## 6. 放行判断口径

目标环境达到当前阶段放行口径，至少需要满足：

- `500,000` 与 `1,000,000` 单任务正式核算均成功
- 高并发任务组合无重复派发、无任务丢失、无分片 owner 冲突
- 长稳压测过程中无持续性积压、无任务卡死、无结果台账异常
- 压测证据可回放，可按 `request_no / task_id` 查到任务、分片、结果台账三层信息

以下情况视为未通过：

- 任务状态停留在 `RUNNING / INIT` 且无法自动恢复
- 分片 owner 分布与任务执行结果不一致
- 同一场景同一账期压测出现异常冲突且无法解释
- 压测产物缺失，无法重建结论
