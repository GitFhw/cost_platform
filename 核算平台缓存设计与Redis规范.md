# 核算平台缓存设计与Redis规范

## 1. 文档目标

本规范用于明确 `cost_platform` 中哪些数据应该进入 Redis、哪些数据不能依赖 Redis、缓存如何失效、哪些链路必须使用 Redis 锁。

Redis 在核算平台中的定位不是“把查得慢的都塞进去”，而是：

- 提升高频读性能
- 承载短时状态
- 支撑分布式锁和幂等
- 支撑运行链路中的临时协同

核算平台的**真实业务数据来源永远是数据库和发布快照**，不是 Redis。

## 2. 缓存设计总原则

### 2.1 缓存不是事实来源

以下对象不能只存在 Redis：

- 场景
- 费用
- 变量
- 规则
- 阶梯
- 发布版本
- 核算结果
- 审计记录

这些对象必须以数据库为准，Redis 只能做加速层。

### 2.2 只缓存高频读、低写频、可重建的数据

优先进入 Redis 的对象，应满足以下条件：

- 读取频繁
- 写入频率较低
- 丢失后可从数据库或快照重建
- 对一致性要求允许短暂延迟

### 2.3 发布链路优先保证一致性

与发布、生效、回滚、正式核算强相关的链路，不允许因为 Redis 缓存脏数据影响业务正确性。

这些链路必须遵循：

- 关键判断先读数据库/快照
- Redis 只做辅助索引或只读加速
- 发布完成后主动失效相关缓存

### 2.4 前端不保留事实缓存

- 页面列表、详情、下拉、指标卡默认查后台接口
- 如需命中 Redis，也必须由后端统一封装后返回，前端不直接维护共享缓存
- 禁止在前端 store、localStorage、sessionStorage 中长期缓存字典、场景、费用、规则等事实数据并跨页面复用
- 前端本地状态只用于当前渲染过程，不作为多人协作场景下的一致性来源

## 3. 适合放入 Redis 的数据

## 3.1 系统字典缓存

适合缓存内容：

- 业务域字典
- 状态字典
- 货种、班次、工种、岗位、币种等基础字典

缓存原因：

- 高频下拉读取
- 变化不频繁
- 丢失后可立即从数据库恢复

推荐 Key：

- `cost:dict:type:{dictType}`
- `cost:dict:data:{dictType}`

推荐 TTL：

- 12 小时到 24 小时

失效策略：

- 字典类型或字典项新增、修改、停用、删除后主动删除对应 Key

## 3.2 场景远程下拉与轻量主数据缓存

适合缓存内容：

- 场景名称、场景编码、业务域、状态这类轻量列表
- 场景远程搜索下拉结果

缓存原因：

- 前端查询频繁
- 列表字段少
- 失效成本低

推荐 Key：

- `cost:scene:summary:list:{keyword}:{status}:{pageNo}:{pageSize}`
- `cost:scene:summary:{sceneId}`

推荐 TTL：

- 5 分钟到 15 分钟

注意：

- 只缓存摘要，不缓存完整业务对象

## 3.3 费用摘要缓存

适合缓存内容：

- 某场景下费用列表摘要
- 费用远程选择器结果
- 费用中心顶部指标卡统计结果

推荐 Key：

- `cost:fee:summary:scene:{sceneId}`
- `cost:fee:metric:scene:{sceneId}:{queryHash}`

推荐 TTL：

- 5 分钟到 10 分钟

## 3.4 发布版本元数据缓存

适合缓存内容：

- 场景当前生效版本号
- 场景当前生效版本主键
- 最近发布版本摘要

缓存原因：

- 运行入口、试算入口、结果追溯入口都会高频读取

推荐 Key：

- `cost:publish:active:scene:{sceneCode}`
- `cost:publish:latest:scene:{sceneCode}`

推荐 TTL：

- 不依赖自然 TTL，采用“写后主动覆盖 + 版本变更主动失效”

注意：

- 当前生效版本信息可以缓存在 Redis
- 但核算执行真正装载时，必须以数据库中生效版本记录或发布快照为准

## 3.5 发布快照装载缓存

适合缓存内容：

- 某场景某发布版本的运行时快照对象
- 规则树、变量树、阶梯明细的运行态装配结果

缓存原因：

- 正式核算、试算、批量任务会重复装载同一版本快照
- 每次从数据库拼装成本高

推荐 Key：

- `cost:runtime:snapshot:{sceneCode}:{versionNo}`

推荐 TTL：

- 30 分钟到 2 小时

注意：

- Redis 中缓存的是“快照装配结果”
- 数据库中必须保留正式快照主记录和快照明细，Redis 丢失后可重建

## 3.6 公式模板与表达式辅助缓存

适合缓存内容：

- 平台公式模板
- 常用函数说明
- 场景变量映射摘要

推荐 Key：

- `cost:formula:template:list`
- `cost:formula:context:scene:{sceneId}`

推荐 TTL：

- 30 分钟到 6 小时

## 3.7 查询统计与看板指标缓存

适合缓存内容：

- 顶部卡片统计
- 费用中心统计
- 规则中心统计
- 发布中心统计
- 台账筛选后的轻量聚合

推荐 Key：

- `cost:metric:{module}:{queryHash}`

推荐 TTL：

- 1 分钟到 5 分钟

说明：

- 聚合统计适合缓存
- 明细表格数据仍以数据库分页查询为主

## 3.8 导入预览与短期草稿上下文

适合缓存内容：

- 导入预览结果
- 导入校验结果
- 试算临时输入
- 公式构建器临时上下文

推荐 Key：

- `cost:import:preview:{token}`
- `cost:simulation:temp:{userId}:{sceneCode}:{token}`

推荐 TTL：

- 30 分钟

注意：

- 这类数据天然是临时态，适合 Redis

## 3.9 任务状态与进度缓存

适合缓存内容：

- 正式核算任务实时进度
- 批量任务进度条
- 失败重试次数
- 实时日志游标

推荐 Key：

- `cost:task:progress:{taskNo}`
- `cost:task:runtime:{taskNo}`

推荐 TTL：

- 任务执行中常驻
- 任务完成后 1 到 3 天过期

## 4. 必须使用 Redis 锁的链路

以下链路必须加分布式锁，避免重复执行或状态冲突：

### 4.1 场景发布

推荐 Key：

- `cost:lock:publish:scene:{sceneCode}`

原因：

- 避免同一场景同时发布两个版本

### 4.2 场景生效切换

推荐 Key：

- `cost:lock:activate:scene:{sceneCode}`

原因：

- 避免多次切换生效版本导致状态错乱

### 4.3 正式核算单据执行

推荐 Key：

- `cost:lock:calc:scene:{sceneCode}:{bizNo}`

原因：

- 避免重复计算同一业务单据

### 4.4 批量任务启动

推荐 Key：

- `cost:lock:batch:{sceneCode}:{batchNo}`

原因：

- 避免同一批次重复启动

### 4.5 导入处理

推荐 Key：

- `cost:lock:import:{sceneCode}:{fileHash}`

原因：

- 避免同一文件重复导入

## 5. 必须使用 Redis 做幂等控制的场景

### 5.1 发布幂等

Key：

- `cost:idempotent:publish:{sceneCode}:{requestNo}`

### 5.2 正式核算幂等

Key：

- `cost:idempotent:calc:{sceneCode}:{bizNo}:{requestNo}`

### 5.3 批量任务幂等

Key：

- `cost:idempotent:batch:{sceneCode}:{batchNo}:{requestNo}`

## 6. 不建议放入 Redis 的数据

以下数据不应作为 Redis 主体承载：

- 草稿配置完整对象
- 审计日志
- 发布快照的唯一事实数据
- 正式核算结果明细
- 财务对账结果
- 长期台账

原因：

- 这类数据必须长期可追溯、可审计、可恢复
- Redis 不适合作为唯一存储

## 7. 缓存失效策略

### 7.1 主动失效优先

以下操作完成后必须主动删除相关缓存：

- 场景新增、修改、停用、删除
- 费用新增、修改、停用、删除
- 变量新增、修改、停用、删除
- 规则新增、修改、停用、删除
- 阶梯新增、修改、删除
- 发布、生效、回滚
- 字典项修改

### 7.2 TTL 只做兜底

缓存不能只靠 TTL 等自然过期。

推荐策略：

- 数据变更时主动失效
- TTL 作为兜底，防止脏缓存永久存在

## 8. Key 命名规范

统一格式：

`cost:{模块}:{子模块}:{对象}:{业务键}`

示例：

- `cost:dict:data:cost_business_domain`
- `cost:scene:summary:3`
- `cost:fee:summary:scene:8`
- `cost:publish:active:scene:salary-company-a`
- `cost:runtime:snapshot:salary-company-a:V2026.03`
- `cost:task:progress:TASK20260330001`

## 9. 序列化规范

### 9.1 推荐序列化格式

- 字典和摘要对象：JSON
- 运行时快照：JSON 或压缩 JSON
- 锁与幂等标识：字符串

### 9.2 不建议直接缓存 ORM 对象

缓存内容应为明确的缓存 DTO，而不是数据库实体对象，避免：

- 结构膨胀
- 字段误用
- 版本兼容性差

## 10. 监控与告警

Redis 使用必须纳入监控：

- 命中率
- 平均加载耗时
- 大 Key
- 热 Key
- 锁等待时长
- 幂等冲突次数

对以下情况建议告警：

- 生效版本缓存与数据库不一致
- 运行时快照缓存频繁失效
- 场景发布锁长时间未释放
- 批量任务进度缓存异常丢失

## 11. 研发执行要求

后续开发必须遵守：

- 先定义缓存对象，再落代码
- 先定义 Key，再实现 Redis 操作
- 关键链路必须写清楚缓存命中、失效和锁逻辑
- Redis 使用必须在代码注释中说明业务目的

## 12. 后续推荐的缓存接入顺序

第一阶段接入：

- 字典缓存
- 场景远程下拉缓存
- 费用摘要缓存

第二阶段接入：

- 发布版本元数据缓存
- 快照装载缓存
- 发布锁与幂等

第三阶段接入：

- 批量任务进度缓存
- 试算临时上下文
- 看板聚合缓存
